
# Brainstorm

## Versão 1.0

## Perguntas

### 1. Qual o objetivo principal da aplicação?

<p align="justify">
<b>João</b> - Deve ser uma plataforma onde administradores consigam cadastrar eventos internos e automatizar a inscrição de alunos.

<b>Maria</b> - A plataforma deve fornecer um sistema simples e rápido para organizar eventos acadêmicos ou corporativos.

<b>Carlos</b> - O objetivo da aplicação é reduzir o trabalho manual de inserir alunos em eventos.

<b>Ana</b> - O principal objetivo é automatizar a alocação de alunos com base em critérios definidos.

<b>Lucas</b> - A plataforma deve gerenciar eventos, inscrições e listas de participantes de forma eficiente.
</p>

---

### 2. Como será o processo para cadastrar um novo cliente (usuário)?

<p align="justify">
<b>João</b> - O administrador deverá fazer login e cadastrar novos usuários manualmente ou por importação de planilha.

<b>Maria</b> - O usuário pode se cadastrar utilizando e-mail institucional.

<b>Ana</b> - Com o usuário logado, ele poderá completar seu perfil com curso, turma e interesses.

<b>Carlos</b> - O sistema pode permitir login integrado com Google ou sistema acadêmico.

<b>Lucas</b> - O cadastro pode incluir níveis de acesso: administrador, organizador e aluno.
</p>

---

### 3. Como será a forma de adicionar eventos?

<p align="justify">
<b>João</b> - O administrador poderá cadastrar eventos com nome, data, vagas e descrição.

<b>Maria</b> - O evento deve ter critérios de participação (curso, período, etc.).

<b>Carlos</b> - O sistema pode permitir upload de listas de alunos para inscrição automática.

<b>Ana</b> - Deve ser possível configurar eventos com inscrição manual ou automática.

<b>Lucas</b> - O evento pode ter limite de vagas e lista de espera.
</p>

---

### 4. Outras perguntas pertinentes ao contexto

<p align="justify">
<b>João</b> - O sistema pode enviar notificações quando o aluno for inscrito em um evento.

<b>Maria</b> - O aluno deve conseguir visualizar todos os eventos disponíveis.

<b>Carlos</b> - O sistema pode gerar relatórios de participação.

<b>Ana</b> - Deve haver controle de presença no evento.

<b>Lucas</b> - Integração com calendário (Google Agenda) seria interessante.
</p>

---

### 5. Como seria a forma do cliente (administrador) adicionar alunos aos eventos?

<p align="justify">
<b>João</b> - O administrador pode importar uma lista de alunos via CSV.

<b>Maria</b> - O sistema pode sugerir alunos automaticamente com base em critérios.

<b>Carlos</b> - Deve ser possível adicionar alunos manualmente.

<b>Ana</b> - Pode haver regras automáticas para inscrição (ex: todos de um curso).

<b>Lucas</b> - O sistema pode priorizar alunos que ainda não participaram de eventos.
</p>

---

### 6. Quais informações seriam interessantes para o usuário?

<p align="justify">
<b>João</b> - Informações sobre eventos disponíveis, datas e vagas restantes.

<b>Maria</b> - O usuário poderá acessar histórico de participação.

<b>Carlos</b> - Informações sobre certificados e presença.

<b>Ana</b> - Detalhes do evento como local, organizador e requisitos.

<b>Lucas</b> - O usuário poderá ver notificações, status de inscrição e lista de espera.
</p>

---

# Conclusão
<p align="justify">
Através da aplicação da técnica de brainstorm, foi possível identificar ideias iniciais e levantar requisitos importantes para o desenvolvimento do sistema de automação de inserção de alunos em eventos internos. Essas informações servirão como base para as próximas etapas do projeto, como modelagem e prototipação.
</p>
